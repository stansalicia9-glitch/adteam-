# 系统逻辑

本文档说明 FF_team 的核心数据流和操作逻辑，仅记录系统机制，不记录任何真实账号、密码、Token、Cookie、API Key 或代理信息。

## 主程序目录

当前主程序目录：

```text
C:\Users\Administrator\Desktop\aeobe\1.0\FF_team_dist\FF_team
```

Web 入口默认端口为 `5005`，由 `start.bat` 启动。

## 核心对象

- 母号：Adobe 团队管理员账号。
- 子号：加入母号团队并用于导出 CK 的成员账号。
- 分组：给母号打的业务分组，用于批量登录、删加、导出、推送隔离。
- CK：子号登录 Adobe Firefly 后导出的 Cookie 字符串。

## 主要数据文件

- `admin_console_config.json`
  - 保存母号配置、分组、席位、产品地址、org/product、JIL token 等。
  - 新导入母号可以指定分组。

- `console_children.json`
  - 保存每个母号当前对应的子号清单。
  - 这是“当前有效子号”的依据。

- `firefly_adobe2api_cookies.json`
  - 保存已经成功导出的子号 CK。
  - 如果这个文件被清空，本地就没有已导出的 CK，需要重新导出。

- `firefly_mail_pool_state.json`
  - 保存邮箱池状态。
  - 子号删加时会从邮箱池预占新账号。

- `push_config.json`
  - 保存推送 API 默认配置和分组覆盖配置。
  - 分组有单独配置时优先使用分组配置，否则回落默认配置。

- `push_records.json`
  - 保存推送记录。
  - 用于排查每次推送的母号、分组、数量、HTTP 状态、job_id 和错误。

- `admin_profile\`
  - 保存每个母号独立的浏览器登录态。
  - 每个母号一个独立 profile，目录名由母号邮箱转换而来。

## 母号登录逻辑

母号登录用于获取 Adobe Admin Console 的组织信息和管理 token。

流程：

1. 打开母号独立浏览器 profile。
2. 如果已有可用 session，则尝试复用。
3. 如果没有可用 session，则自动输入账号、密码、邮箱验证码。
4. 选择企业 profile。
5. 进入 Admin Console。
6. 提取 `org_id`、`product_id`、`product_users_url`、`jil_token`。
7. 成功提取后才标记为已播种。

注意：

- 只打开到 Admin Console 根页不等于登录完全成功。
- 只有成功提取到组织和 token 后，才算母号登录闭环完成。
- 如果 profile 脏了，可以清除该母号的 `admin_profile` 目录后重新登录。

## 删加子号逻辑

删加子号按母号执行。

流程：

1. 读取母号的 org/product/token。
2. 查询当前产品用户。
3. 保留管理员母号。
4. 删除非管理员旧子号。
5. 按席位数从邮箱池预占新子号。
6. 添加新子号到 Adobe 团队。
7. 成功后写入 `console_children.json`。

说明：

- `added_accounts.txt` 是历史添加日志，不再作为当前子号依据。
- 当前子号以 `console_children.json` 为准。

## CK 导出逻辑

CK 导出用于登录子号并导出可供外部程序使用的 Cookie。

支持三种入口：

- 单个母号导出当前子号 CK。
- 勾选多个母号后导出选中母号当前子号 CK。
- 底部导出全部母号当前子号 CK。

分组行为：

- 如果页面选择了分组，批量导出默认只处理当前分组。
- 如果手动勾选了母号，则以勾选项为准。

导出结果：

- 成功导出的 CK 写入 `firefly_adobe2api_cookies.json`。
- 某个子号失败时，该子号不会有 CK。
- 一个母号 9 个子号只成功 7 个，则本地只保存 7 个 CK。

## 推送逻辑

推送 API 是 CK 导出后的旁路功能，不影响 CK 导出本身。

流程：

1. 母号导出任务结束。
2. 按该母号当前子号列表，从 `firefly_adobe2api_cookies.json` 筛选已成功导出的 CK。
3. 如果至少有 1 个 CK，则按母号推送一次。
4. 如果 0 个 CK，则跳过推送并写记录。
5. 推送成功/失败都会写入 `push_records.json`。

推送配置选择：

1. 如果母号所属分组有单独推送配置，使用分组配置。
2. 如果该分组没有配置，使用默认推送配置。
3. 如果默认配置也未启用或不完整，则不推送。

推送失败不会影响：

- CK 导出结果。
- `firefly_adobe2api_cookies.json`。
- 后续母号导出。

## 分组逻辑

母号可以设置分组。

分组用途：

- 筛选母号列表。
- 按组批量登录。
- 按组批量扫描。
- 按组批量删加子号。
- 按组导出当前子号 CK。
- 按组使用不同推送 API 配置。

导入新母号时可以填写“导入到分组”，新母号会自动写入该分组。

## 常见状态说明

- 已播种：母号已经成功登录并提取过 org/token。
- 未播种：母号还没有完成登录提取。
- 当前子号：该母号最近一次成功删加后记录的子号数量。
- 已导 Cookie：本地 CK 库中成功保存的 CK 数量。
- 待导 Cookie：当前系统认为还缺少 CK 的账号数量。

## 安全注意事项

- 不要把真实 `admin_console_config.json`、`push_config.json`、`firefly_adobe2api_cookies.json`、邮箱池文件直接发给别人。
- 发包前应清空母号、子号、邮箱池、CK、推送配置、推送记录和浏览器 profile。
- `firefly_adobe2api_cookies.json` 清空后，本地 CK 不可恢复，只能重新导出。
